# План поэтапного внедрения AttributeType через ModelFactory (без затрагивания UI)

## Глобальные ограничения

* ❗ НЕЛЬЗЯ изменять `ActiveField`, формы и рендер на первых этапах
* ❗ Весь переход начинается только через `ModelFactory` и генераторы
* ❗ В `attributeData()` добавляется **новое** поле `typeClass`, не влияющее на текущий код
* ❗ Каждый этап закрывается тестами
* ❗ Переход к следующему этапу только после прохождения тестов

---

## Этап 1 — Введение `typeClass` в metadata атрибутов

### Цель

Добавить поддержку нового поля в `attributeData()` без использования его где-либо.

### Что сделать

1. В моделях разрешить структуру:

```php
[
    'label' => '...',
    'hint' => '...',
    'typeClass' => StringType::class,
]
```

2. Ничего в проекте не должно это поле использовать.

### Тесты этапа

Написать тест:

> Проверяет, что `attributeData()` может содержать `typeClass` и это не ломает модель, валидацию и рендер форм.

Критерий: формы и текущие тесты работают без изменений.

---

## Этап 2 — Базовый AttributeTypeInterface и 2–3 простых типа

### Цель

Ввести систему типов, но использовать её нигде.

### Что сделать

1. Создать `AttributeTypeInterface`
2. Реализовать базовые типы:

   * `StringType`
   * `TextType`
   * `DateType`

Пока реализовать только:

```php
name()
generator()
samples()
apiSchema()
```

Без renderInput / renderOutput.

### Тесты этапа

Тест, который:

* создаёт экземпляры типов
* проверяет `samples()`, `apiSchema()`, `generator()` возвращают валидные данные

---

## Этап 3 — Доступ к типу из модели (ТОЛЬКО для генерации)

### Цель

Модель умеет вернуть тип, но никто кроме генерации его не использует.

### Что сделать

Добавить в `ArmsModel`:

```php
public function getAttributeTypeForGeneration(string $attribute): ?AttributeTypeInterface
```

Логика:

* если в `attributeData()['typeClass']` есть класс — вернуть экземпляр
* иначе вернуть `null`

### Тесты этапа

Тест:

* для модели с `typeClass` возвращается объект типа
* для модели без `typeClass` возвращается `null`

---

## Этап 4 — Интеграция с GeneratorResolver (с fallback)

### Цель

GeneratorResolver начинает использовать типы, если они есть.

### Что сделать

В `GeneratorResolver`:

```php
$type = $model->getAttributeTypeForGeneration($attribute);

if ($type && $type->generator()) {
    использовать generator типа
} else {
    использовать старую логику
}
```

### Тесты этапа

1. Для атрибута с `typeClass` используется генератор типа
2. Для атрибута без `typeClass` используется старый генератор
3. Обе модели успешно генерируются и проходят валидацию

---

## Этап 5 — Генеративный тест покрытия моделей типами

### Цель

Начать выявлять, где нет `typeClass`, но пока не падать.

### Что сделать

Написать тест:

> Проходит по всем моделям и всем записываемым атрибутам
> Если нет `typeClass` — пишет warning (не fail)

### Тесты этапа

Тест успешно проходит и выводит список атрибутов без типа.

---

## Этап 6 — Массовое заполнение моделей `typeClass`

### Цель

Покрыть большинство атрибутов типами.

### Что сделать

Постепенно добавлять `typeClass` в модели.

### Тесты этапа

Генеративный тест показывает уменьшение warning.
Все модели успешно генерируются и валидируются.

---

## Этап 7 — Типы становятся источником правды

### Цель

Начать использовать в типах:

* `gridColumnClass`
* `samples`
* `apiSchema`

Но UI всё ещё не трогаем.

### Тесты этапа

Тесты на:

* корректность `apiSchema`
* корректность `samples`
* что тип возвращает валидный `gridColumnClass`

---

## Этап 8 — Подготовка к UI (ничего не ломая)

### Цель

Добавить в типы методы:

```php
renderInput()
renderOutput()
```

Но пока их нигде не вызывать.

### Тесты этапа

Тест, который вручную вызывает `renderInput()` у типов и проверяет, что возвращается строка HTML.

---

## Этап 9 — Подключение ActiveField к типам с fallback

⚠️ Только после полного покрытия тестами.

### Что сделать в ActiveField

```php
$type = $model->getAttributeTypeForGeneration($attribute);

if ($type) {
    render через тип
} else {
    старая логика
}
```

### Тесты этапа

Тест рендера форм:

* поля с типом рендерятся
* поля без типа рендерятся по-старому

---

## Этап 10 — Удаление fallback

### Цель

Сделать `typeClass` обязательным.

Вместо fallback:

```php
throw new Exception("typeClass not defined");
```

### Тесты этапа

Генеративный тест и тест рендера форм должны полностью проходить без fallback.

---

# Результат

К моменту изменения UI:

* Типы уже протестированы генерацией
* Модели полностью покрыты
* Риск поломки форм минимален
* ActiveField становится тонким слоем без бизнес-логики
