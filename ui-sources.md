# UI ← источники данных и документации

Обратная карта: для каждого элемента интерфейса — откуда берутся его тексты
и оформление. Прямой взгляд (формат `attributeData()` со всеми ключами) описан
в докблоке [`AttributeDataModelTrait`](models/base/traits/AttributeDataModelTrait.php);
конвенция ведения документации — в [docs/help/README.md](docs/help/README.md);
план — в [plans/help-docs.md](plans/help-docs.md).

Источник правды двуслойный:

- **слой 1 (код)**: `attributeData()` модели, статики/методы модели
  (`$title`, `$titles`, `modelDescription()`), методы «толстого» типа
  (`inputHint()`, `searchHint()`, `renderInput()`, `gridColumnClass()`,
  `apiSchema()`);
- **слой 2 (MD-файлы)**: `docs/help/models/<class-id>.md`,
  `docs/help/models/<class-id>/<attr>.md`, `docs/help/types/<type>.md`
  (+ каталог переопределений заказчика `params['docsOverridePath']`).

Обозначение `a → b → c` — порядок фолбэка (берётся первое непустое).

---

## 0. Вклад типов в UI (сводно)

«Толстый» тип атрибута — полноправный источник лёгкого слоя: его тексты
подклеиваются к текстам атрибута везде, где атрибут показывается.

| Метод типа | Где проявляется |
|---|---|
| `inputHint()` — «как вводить» | тултип label в форме: приглушённо после hint атрибута |
| `searchHint()` — «как искать» | тултип колонки с фильтром: приглушённо после общего синтаксиса поиска |
| `renderInput()` | виджет ввода в форме (через `autoInput()`) |
| `gridColumnClass()` | класс/рендер колонки в списках |
| `apiSchema()`, `samples()` | Swagger/REST API |
| `types/<type>.md` (слой 2) | ссылка «подробнее о типе: … »» из тултипа атрибута; ссылка со страницы `/docs/model` |

Как эти источники собираются в конечный тултип — см. следующий раздел,
он является спецификацией для кода.

## 0.1 Механизм сборки тултипа атрибута (спецификация)

### Принцип

Тултип атрибута собирается **в одной точке** — компонент-сборщик
(`AttributeTooltip`, этап 3) с сигнатурой:

```
build(ArmsModel $model, string $attr, string $mode): ?array  // [title, body]
```

Режимы: `form` (label поля), `grid` (заголовок колонки), `search`
(заголовок колонки с фильтром), `view` (карточка объекта).
Все потребители — `ActiveField::renderLabelParts()`,
`FieldsHelper::labelOption()`, `AttributeHintWidget` (grid/search/view),
`views/docs/model.php` — зовут сборщик и ничего не клеят сами.

`getAttributeHint()` при этом возвращает **чистую смысловую часть**
(только attributeData, без типовых довесков) — композиция уходит из
уровня данных в сборщик. Макрос `{same}` везде подставляет чистый hint.
(Сейчас, на этапе 2, композиция временно живёт в `getAttributeHint()`,
из-за чего формат ввода просачивается в grid/карточку — сборщик это
устраняет.)

### Блоки тела тултипа

Тело — последовательность блоков. Блоки разделяются `<hr/>`,
строки внутри блока — `<br>`. Пустые блоки пропускаются.

| # | Блок | Содержимое | form | grid | search | view |
|---|---|---|:-:|:-:|:-:|:-:|
| 1 | Смысл («что это») | по режиму: `hint` (form); `indexHint → viewHint → hint` (grid/search); `viewHint → indexHint → hint` (view); `{same}` = чистый hint | ✓ | ✓ | ✓ | ✓ |
| 1а | Формат («как вводить») — продолжение блока 1 через `<br>`, обёрнут `<span class="text-muted">` | `inputHint()` типа | ✓ | — | — | — |
| 2 | Поиск («как искать») | явный `searchHint` из attributeData (`{same}` поддерживается) — **если задан, блок состоит только из него**; иначе: дефолт по типу данных (строка/число/дата из `apiSchema()`) `<br>` + muted `searchHint()` типа | — | — | ✓ | — |
| 3 | Переходы на слой 2 | ссылка «подробнее: <Titles> → <Label> »» ← `models/<class-id>/<attr>.md`; ссылка «подробнее о типе: <H1 страницы> »» ← `types/<type>.md`; каждая — только если файл существует, по одной на строку, класс `open-in-modal-form` | ✓ | ✓ | ✓ | ✓ |

Заголовок тултипа (card-header) — label того же режима:
`label` (form), `indexLabel → viewLabel → label` (grid/search),
`viewLabel → label` (view).

### Условия отображения

- Все блоки пусты → сборщик возвращает `null`: в форме не выводится
  иконка «?», в grid/view label рендерится без тултип-обёртки.
- `hint` пуст, но тип даёт `inputHint()` → блок 1 состоит только из
  muted-формата (тултип есть).
- Тип атрибута не выводится (вычисляемое поле) → блока 1а нет;
  в блоке 2 дефолт — строковый.
- Явный `searchHint` в attributeData полностью вытесняет и дефолт,
  и типовую часть поиска (уже закреплено тестом
  `TypeHintCompositionTest::testExplicitSearchHintWins`).

### Канон разметки

Тултип — bootstrap-карточка (как в `FieldsHelper::toolTipOptions()`):

```html
<div class="card">
  <div class="card-header">{title}</div>
  <div class="card-body"><p class="card-text">{body}</p></div>
</div>
```

qtip-атрибуты: `qtip_ttip` (контент), `qtip_side`, `qtip_theme`;
тултипы интерактивные (`interactive: true`) — ссылки блока 3 кликабельны.

## Шаблоны описаний

Цель — чтобы конкатенации (атрибут + тип) и переходы между слоями читались
как один связный текст. Каждый текст отвечает ровно на свой вопрос и не
залезает на территорию соседа.

### hint атрибута (`attributeData`) — «ЧТО здесь хранится»

- 1–3 предложения, до ~300 символов, допустима лёгкая разметка
  (`<br>`, `<b>`, `<i>`); заголовки/списки разделов — признак, что тексту
  место в слое 2.
- Не описывает формат ввода (его даст `inputHint()` типа), не повторяет label.
- Шаблон: `<Что хранится/означает>. [<На что влияет / когда заполнять>]`

```php
//хорошо: только смысл поля в контексте модели
'hint' => 'MAC адреса сетевых интерфейсов, настроенных в ОС',
//плохо: формат — территория типа, будет продублирован
'hint' => 'MAC адреса. Заполняются по одному в строке',
```

### `modelDescription()` — «ЧТО это за сущность»

- 1–3 предложения: что это за объекты, что учитывается, ключевые связи.
- Без инструкций (они в guides) и перечисления всех полей (оно генерируется).

```php
public static function modelDescription(): string
{
	return 'Единицы оборудования: серверы, АРМ, сетевые устройства, периферия. '
		.'Учитываются модель, серийный/инвентарный номера, размещение, состояние '
		.'и связи с ОС, пользователями и документами.';
}
```

### `inputHint()` типа — «КАК вводить»

- Начинается с существа формата, без вводных («В каждой строке — …»,
  «Формат: …», «Выберите из списка…»).
- Не упоминает конкретные модели/атрибуты — текст один на все места
  использования типа.
- Выводится приглушённо (`text-muted`) после хинта атрибута — вместе они
  читаются как «что это» + «как заполнять».

### `searchHint()` типа — «КАК искать»

- Только отличия/дополнения к общему синтаксису поиска (`|`, `&`, `!` уже
  описаны в дефолтной подсказке, к которой текст подклеивается).

### MD сущности (`docs/help/models/<class-id>.md`)

Только концепция — **без разделов по отдельным атрибутам** (короткое
описание атрибута живёт в hint, длинное — в `models/<class-id>/<attr>.md`;
страница сущности в приложении рендерит атрибутные страницы автоматически
в секции «Подробно об атрибутах»).

```markdown
# <Titles модели>

<Абзац: расширенное назначение — самодостаточное резюме страницы.>

## Как заводится / жизненный цикл
## Связи с другими сущностями
## Типовые операции
```

### MD атрибута (`docs/help/models/<class-id>/<attr>.md`)

```markdown
# <Titles> → <Label атрибута>

<Абзац: когда и как заполняется, на что влияет.>

## Примеры
## Связанное
<опционально ссылка на страницу типа: [тип …](../../types/<type>.md)>
```

### MD типа (`docs/help/types/<type>.md`)

Правила ввода и поиска — территория **лёгкого** слоя (`inputHint()`/
`searchHint()`), на странице типа они не повторяются. Сюда — то, что
в тултип не влезает: устройство, хранение, ограничения, примеры.

```markdown
# <Человеческое название типа>

<Абзац: что за данные, где используются.>

## Хранение и нормализация
## Особенности и ограничения
## Примеры
```

### Guides / Admin (`docs/help/guides|admin/*.md`)

```markdown
# Как <задача пользователя>   /   # <Процедура администрирования>

<Абзац: результат и когда это нужно.>

1. Шаг…
2. Шаг…
```

Общие правила страниц: первая строка — `# Заголовок` (он же title и подпись
ссылок), первый абзац — самодостаточное резюме (в модалке виден первым),
ссылки между страницами относительные, картинки — только когда без них никак.
