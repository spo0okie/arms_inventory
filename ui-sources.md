# UI ← источники данных и документации

Обратная карта: для каждого элемента интерфейса — откуда берутся его тексты
и оформление. Прямой взгляд (формат `attributeData()` со всеми ключами) описан
в докблоке [`AttributeDataModelTrait`](models/base/traits/AttributeDataModelTrait.php);
конвенция ведения документации — в [docs/help/README.md](docs/help/README.md);
план — в [plans/help-docs.md](plans/help-docs.md).

Источник правды двуслойный:

- **слой 1 (код)**: `attributeData()` модели, статики/методы модели
  (`$title`, `$titles`, `modelDescription()`), методы «толстого» типа
  (`inputHint()`, `searchHint()`, `renderInput()`, `gridColumnClass()`,
  `apiSchema()`);
- **слой 2 (MD-файлы)**: `docs/help/models/<class-id>.md`,
  `docs/help/models/<class-id>/<attr>.md`, `docs/help/types/<type>.md`
  (+ каталог переопределений заказчика `params['docsOverridePath']`).

Обозначение `a → b → c` — порядок фолбэка (берётся первое непустое).

---

## 0. Вклад типов в UI (сводно)

«Толстый» тип атрибута — полноправный источник лёгкого слоя: его тексты
подклеиваются к текстам атрибута везде, где атрибут показывается.

| Метод типа | Где проявляется |
|---|---|
| `inputHint()` — «как вводить» | тултип label в форме: приглушённо после hint атрибута |
| `searchHint()` — «как искать» | тултип колонки с фильтром: приглушённо после общего синтаксиса поиска |
| `renderInput()` | виджет ввода в форме (через `autoInput()`) |
| `renderOutput()` — «как показать значение» | значение атрибута везде, где он выводится через `ModelFieldWidget` (карточки, DetailView, grid через `DefaultColumn`); контракт см. §3 |
| `gridColumnClass()` | класс/рендер колонки в списках |
| `apiSchema()`, `samples()` | Swagger/REST API |
| `types/<type>.md` (слой 2) | ссылка «подробнее о типе: … »» из тултипа атрибута; ссылка со страницы `/docs/model` |

Как эти источники собираются в конечный тултип — см. следующий раздел,
он является спецификацией для кода.

## 0.1 Механизм сборки тултипа атрибута (спецификация)

### Принцип

Тултип атрибута собирается **в одной точке** — компонент-сборщик
(`AttributeTooltip`, этап 3) с сигнатурой:

```
build(ArmsModel $model, string $attr, string $mode): ?array  // [title, body]
```

Режимы: `form` (label поля), `grid` (заголовок колонки), `search`
(заголовок колонки с фильтром), `view` (карточка объекта).
Все потребители — `ActiveField::renderLabelParts()`,
`FieldsHelper::labelOption()`, `AttributeHintWidget` (grid/search/view),
`views/docs/model.php` — зовут сборщик и ничего не клеят сами.

`getAttributeHint()` при этом возвращает **чистую смысловую часть**
(только attributeData, без типовых довесков) — композиция живёт в
сборщике. Макрос `{same}` везде подставляет чистый hint.

### Блоки тела тултипа

Тело — последовательность блоков. Блоки разделяются `<hr/>`,
строки внутри блока — `<br>`. Пустые блоки пропускаются.

| # | Блок | Содержимое | form | grid | search | view |
|---|---|---|:-:|:-:|:-:|:-:|
| 1 | Смысл («что это») | по режиму: `hint` (form); `indexHint → viewHint → hint` (grid/search); `viewHint → indexHint → hint` (view); `{same}` = чистый hint | ✓ | ✓ | ✓ | ✓ |
| 1а | Формат («как вводить») — продолжение блока 1 через `<br>`, обёрнут `<span class="text-muted">` | `inputHint()` типа | ✓ | — | — | — |
| 1б | Источник значения («откуда взялось») — продолжение блока 1 через `<br>`, обёрнут `<span class="text-muted">` | автоматически из модели, руками не пишется. Порядок проверки: **собирательный** атрибут (`is_collectable` — значение собирается с записи и её ПОТОМКОВ; объявляется на самом Recursive-атрибуте, сторож `RecursiveAttrDirectionTest`) — «собирательный атрибут: значение собирается с этой записи и её потомков»; **наследуемый** (`is_inheritable` либо конвенция суффикса `<attr>Recursive` — направление «вверх», как в `ArmsModel::__get`; алиасы link↔getter разрешаются) — приоритет над «вычисляемым» (унаследованное значение тоже вычисляется, наследуемость конкретнее): у загруженной записи — «наследуемый атрибут: значение задано в этой записи / унаследовано от <renderItem предка — правило unification.md «имя объекта — всегда renderItem»> / значение не задано ни здесь, ни у предков», у пустой — общая пометка «наследуемый атрибут»; прочие не-колонки — «вычисляемое поле»; хранимые колонки и ссылочные атрибуты блока не дают | — | — | — | ✓ |
| 2 | Поиск («как искать») | явный `searchHint` из attributeData (`{same}` поддерживается) — **если задан, блок состоит только из него**; иначе: дефолт по типу данных (строка/число/дата из `apiSchema()`) `<br>` + muted `searchHint()` типа | — | — | ✓ | — |
| 3 | Переходы на слой 2 | ссылка «подробнее: <Titles> → <Label> »» ← `models/<class-id>/<attr>.md`; ссылка «подробнее о типе: <H1 страницы> »» ← `types/<type>.md`; каждая — только если файл существует, по одной на строку, класс `open-in-modal-form` | ✓ | ✓ | ✓ | ✓ |

Заголовок тултипа (card-header) — label того же режима:
`label` (form), `indexLabel → viewLabel → label` (grid/search),
`viewLabel → label` (view).

### Условия отображения

- Все блоки пусты → сборщик возвращает `null`: иконка «?» не выводится
  ни в одном режиме, label рендерится чистым.
- `hint` пуст, но тип даёт `inputHint()` → блок 1 состоит только из
  muted-формата (тултип есть).
- Тип атрибута не выводится (вычисляемое поле) → блока 1а нет;
  в блоке 2 дефолт — строковый.
- Явный `searchHint` в attributeData полностью вытесняет и дефолт,
  и типовую часть поиска (уже закреплено тестом
  `TypeHintCompositionTest::testExplicitSearchHintWins`).
- Интроспекция цепочки наследования в блоке 1б требует загруженной
  записи (зависит от конкретного объекта); у пустой модели (например,
  страница `/docs/model`) наследуемые виртуальные атрибуты получают
  общую пометку «наследуемый атрибут» — приоритет наследуемости
  над «вычисляемым» сохраняется и без записи.
- Блок 1б — единственный источник аннотации наследуемости в UI:
  потребители (в т.ч. `ModelFieldWidget::fieldTitle`) ничего
  не дописывают сами.

### Канон разметки

Подача: тултип атрибута висит НЕ на label, а на иконке «?» рядом с ним
(по образцу иконки помощи страницы `HintIconWidget`). Иконку рендерит
единственная точка — `AttributeTooltip::icon()`; потребители
(`ActiveField::renderLabelParts()`, `FieldsHelper::labelOption()`,
`AttributeHintWidget`, `ModelFieldWidget::fieldTitle()`) дописывают её
к label и ничего не вешают на сам label. `build()` вернул `null` —
иконки нет, label чистый:

```html
{label} <span class="attr-hint-icon" qtip_pin="1" qtip_ttip="{card}"
  qtip_side="…" qtip_theme="…"><i class="fas fa-question-circle"></i></span>
```

qtip-атрибуты и статусные классы — на span-обёртке, НЕ на FA-элементе:
FontAwesome заменяет `<i>` на `<svg>` и пересоздаёт его при изменении
классов, убивая tooltipster-инстанс (по той же причине `HintIconWidget`
вешает тултип на `<a>`).

Внутри тултипов объекта (ttip-действия `ttip`/`ttip-hw`/`ttips`) `icon()`
возвращает пустую строку — иконку «?» там не показываем (тултип в тултипе —
шум, место компактное). По той же логике `LinkObjectWidget` не делает ссылку
на объект на его собственных страницах view/ttip (`samePage`). Контекст
определяется по префиксу `action->id`; сторож — `AttributeTooltipTest`.

Поведение иконки (`web/tooltipster/js/qtip_ajax.js`):
наведение — тултип; клик — «запинить»: тултип не закрывается по
mouseout, повторный клик отпускает (включается атрибутом `qtip_pin`).
Статусы цветом (`web/css/qtip.css`): `.attr-hint-icon` — приглушённая,
`:hover` — акцент, `.qtip-pinned` — приколота (отчётливо другой цвет).
Для JS, подставляющего контент тултипа динамически, есть заготовка
`AttributeTooltip::iconTemplate()` (иконка без `qtip_ttip`).
У иконки помощи страницы (`HintIconWidget`) pin не применяется —
её клик открывает документацию.

**Скрытая до help-mode подача** — `AttributeTooltip::icon($tooltip, onlyHelp: true)`
добавляет класс `.attr-hint-icon--onlyhelp` (`web/css/qtip.css`): иконка не видна в
обычном режиме и проступает только при включённом help-mode. Для мест свободной
вёрстки и цепочек, где «?» рядом с каждым звеном была бы шумом, но атрибут должен
оставаться самодокументируемым. Потребители — `ModelFieldWidget::renderFieldValueHinted()`
и `ChainWidget`.

**Тултип составного блока** — `AttributeTooltip::buildComposite($model, $fields, $title,
$intro='Использованы атрибуты:')`: для рендера, собранного из нескольких атрибутов
кастомной логикой (значение одним типом не выразить). Тело перечисляет участвующие
атрибуты и их смысл (`title()`/`meaning()` каждого поля). Потребитель —
`ModelFieldWidget::renderCompositeTitle()`.

Тултип — bootstrap-карточка (как в `FieldsHelper::toolTipOptions()`):

```html
<div class="card">
  <div class="card-header">{title}</div>
  <div class="card-body"><p class="card-text">{body}</p></div>
</div>
```

qtip-атрибуты: `qtip_ttip` (контент), `qtip_side`, `qtip_theme`;
тултипы интерактивные (`interactive: true`) — ссылки блока 3 кликабельны.

### 2.1 Жадная загрузка связей грида (join-аннотации attributeData) — СТАНДАРТ

Каждая колонка грида, обращающаяся к связям модели, обязана объявлять их в
`attributeData` атрибута ключом **`'join'`** (список relation-путей, включая
вложенные: `'join'=>['arm.user','netIps.network']`). Иначе каждая строка грида
загружает свои связи отдельными запросами (N+1).

Потребители аннотаций (собирает `ArmsModel::attributesJoins($columns)` по
фактически отображаемым колонкам, см. `DynaGridWidget::fetchVisibleAttributes`):

| Сценарий | Инструмент |
|---|---|
| Индекс с поисковой моделью | `search($params,$columns)` → `ArmsModel::prepareSearch($columns)`: `$query->with($joins)` + `$filter->joinWith($joins)` (образцы: UsersSearch, AcesSearch, AclsSearch) |
| Грид, который кормится relation-ом (`ArrayDataProvider(['allModels'=>...])`) | `$model->relationForGrid('techs','<gridId>')` — грузит связь с with() по видимым колонкам и кладет результат в relation (образцы: partners/view, maintenance-jobs/view) |
| REST | `BaseRestController` — joinWith по запрошенным атрибутам |

Готчи:
- один список кормит и `with()` (данные), и `joinWith()` (фильтр) — **self-join
  и алиасы недопустимы** (`acl.aces` у Aces уронит фильтр: «Not unique table/alias»);
- условные геттеры (не всегда возвращающие ActiveQuery, напр. `Techs::getItStaff`)
  аннотировать нельзя — `with()` на них падает;
- счетчики и корзинки/замочки в аннотациях НЕ нуждаются: `ArmsModel::loaderCount()`
  и `nonDeletableReverseLinks()` считают связи кэшированным GROUP BY (один запрос
  на класс+связь за весь HTTP-запрос);
- если нужны только **id-шники** связей (сигнатуры, проверки вхождения) — не `*_ids`
  (LinkerBehavior загружает модели связи на каждый экземпляр), а
  `ArmsModel::loaderIds($loader) ?? $this->rel_ids` — один запрос на класс+связь
  (образцы: `Aces::aceSignature()`, колонка «В паспорте» в soft/view);
- **via('relation')-связи в with() перезатирают populated-модели соседних путей**:
  `with(['linkPort.tech','linkTech'])` — жадная загрузка linkTech (via('linkPort'))
  заново популирует linkPort СВЕЖИМИ экземплярами без вложенного tech
  (ActiveRelationTrait::populateRelation:247 вызывает populateRelation via-связи
  на тех же primary-моделях). Порядок решает: `['linkTech','linkPort.tech']` работает —
  но в собранных из аннотаций списках порядок не гарантирован, поэтому правило:
  не аннотировать via('relation')-связь, если рядом есть вложенный путь через её
  junction-связь. Связи через viaTable() безопасны — они читают junction напрямую
  (findJunctionRows) и populated-модели не трогают;
- мелкие справочники (состояния, песочницы, маркеры, сегменты) в горячем коде
  резолвить через `Class::getLoadedItem($id,true)` с fast-path
  `isRelationPopulated()` — сами relation-геттеры не трогать (их использует with());
- все эти кэши (loaderCount/loaderIds/getLoadedItem/identity map findLoaded)
  самосогласованы: afterSave/afterDelete их инвалидируют.

## 3. Карточка объекта (`view`)

Единая точка вывода атрибута — `ModelFieldWidget` (правило unification.md
«атрибут — всегда ModelFieldWidget»; формы вызова: полная подача
`::widget()`, `renderFieldTitle()`, `renderFieldValue()`,
`renderFieldRow()`, значение со скрытой «?» — `renderFieldValueHinted()`,
заголовок составного блока — `renderCompositeTitle()`, строка DetailView —
`detailAttribute()`).

| Элемент | Источник | Код |
|---|---|---|
| Label атрибута | `viewLabel → label` (цепочка view-режима) | `ModelFieldWidget::fieldTitle()` → `AttributeTooltip` |
| Тултип атрибута | сборка по §0.1, режим `view` (смысл + блок 1б «источник значения» + переходы) | `ModelFieldWidget::fieldTitle()` → `AttributeTooltip` |
| Значение атрибута | `renderOutput()` класса типа (`typeClass`), строго: тип не резолвится — рендер падает; текстовые — ntext/markdown/dokuwiki по `params['textFields']` (в `TextType`, `TextFieldWidget` — тонкий делегат) | `ModelFieldWidget::typeRenderedValue()` |
| Значение-ссылка (link/loader/`ref`) | объектный путь: `renderItem()` каждого объекта (правило «имя объекта — всегда renderItem») | `ModelFieldWidget` → `ListObjectsWidget` |
| Пустое значение | рендер типа НЕ вызывается; подача пустоты — `show_empty`/`message_on_empty` (по умолчанию карточка/строка скрыта) | `ModelFieldWidget`/`ListObjectsWidget` |
| Вкладки/блоки wiki | поле `links` объекта — **пользовательский контент**, не документация | `TabsWidget::addWikiLinks()`, `WikiPageWidget` |

**Нестандартная подпись блока** (напр. «Участвует в работе сервисов:» для
поля `services`) задаётся параметром `label`, а НЕ `title`: `title`-строка
отключает сборку тултипа-иконки «?», тогда как `label` переопределяет только
текст подписи, сохраняя иконку и тултип атрибута (`fieldTitle(...,$label)` →
`AttributeTooltip::build(...,$labelOverride)`). `title` оставляем лишь для
`false` (режим «только значение») или подписи, не связанной с атрибутом.

**Составной блок** (значение собрано из НЕСКОЛЬКИХ атрибутов кастомной логикой —
одним типом не выразить): значение остаётся во вьюхе, а заголовок блока даёт
`renderCompositeTitle($model, $fields, $label, $tag)` — чистый label + «?», тултип
которой перечисляет участвующие атрибуты и их смысл (`AttributeTooltip::buildComposite`).
Санкционированный способ сохранить самодокументируемость там, где унификация вывода
значения невозможна (`tag='span'` — для инлайн-подписи внутри существующего заголовка).

**Цепочка положения** (Организация / подразделение / … / должность) — `ChainWidget`:
последовательность сегментов из разных атрибутов и даже разных моделей через единый
разделитель `glue`. Формы сегмента: `{model,field}` (значение + скрытая «?»
через `renderFieldValueHinted`), `{model,field,chain:true}` / `{node,chain:true}`
(разворот предков дерева через `getChain`/`parent`), `{object}`, `{text}`. Объекты
рендерятся в режиме только-просмотр (`static=true` — без иконок правки/удаления).

| `/docs/model?class=…` — страница сущности | `modelDescription()` + `models/<class-id>.md` (если есть) + генерируемая таблица всех атрибутов (label + hint + ссылки) + футер «Подробно об атрибутах» (рендер всех `models/<class-id>/<attr>.md` абзацами) |
| `/docs/page?path=…` — произвольная страница | MD-файл; относительные ссылки переписываются на маршруты docs |

## 8. Обвязка страницы: крошки и справка (layout)

Единообразие всех полноценных страниц обеспечивает `views/layouts/main.php`,
а не каждая вьюха отдельно. Полноценная страница = обычный layout-рендер
(index/view/create/update/кастомные экшены), НЕ частичные `ttip`/`item`/
`async-grid`/embed.

| Элемент | Источник | Код |
|---|---|---|
| Хлебные крошки | если вьюха не задала `$this->params['breadcrumbs']` — **авто-вывод**: действие `index` → `[$this->title]`; прочие → `[ [Плюрал → index], $this->title ]`. Явный трейл во вьюхе перекрывает авто-вывод | `views/layouts/main.php` → `Breadcrumbs` |
| Иконка «?» справки страницы | правый край ленты крошек (`<li class="docs-help-crumb">`), единый тогглер режима справки на списках, карточках и формах | `HintIconWidget` (в `main.php`), `page-header.css` |
| Режим справки (help-mode) | клик по «?» переключает `body.help-mode`: иконки «?» атрибутов выходят в акцент и раз в ~7 c дают две мягкие вспышки (карта покрытия подсказками). Тогглер — независимый vanilla-`<script>` в `main.php` (делегирование на `document`, не зависит от `jQuery.ready`) | `web/css/qtip.css` (`@keyframes attr-hint-heartbeat`), `main.php` |
| Инфопанель документации | секции MD подтягиваются панелью: список — над таблицей (`DynaGridWidget`), карточка/формы — блоком под крошками (`main.php`: view → «Просмотр», update → «Редактирование», create → «Добавление»); тот же «?» её раскрывает, если она есть на странице | `DocsPanelWidget` |
| Подсветка элемента по ссылке (doc-anchor) | ссылка `<a href="#doc-anchor:КЛЮЧ">` скроллит к элементу `data-doc-anchor="КЛЮЧ"` и подсвечивает его. Обработчик делегирован на `document`, поэтому источником может быть **любое место DOM**: MD-страница/панель, обычный hint-тултип атрибута (`attributeData` допускает HTML — кладём ссылку прямо в hint) или другой элемент. Живой пример: hint `Comps.name` → поле «Песочница» в форме | `main.php` (vanilla-`<script>`), `web/css/site.css` (`.doc-anchor-highlight`) |

**Правило вьюхи.** Новые index/view/create-страницы крошки **не задают** —
их собирает layout из класса контроллера, действия и `$this->title`. Явный
`$this->params['breadcrumbs']` пишем ТОЛЬКО для нестандартного трейла:
- `update` — средним звеном ссылка на карточку объекта
  (`[Плюрал → index, имя → view, «Редактирование…»]`);
- деревья/иерархии — цепочка предков;
- любой трейл, не совпадающий с авто-выводом 1-в-1.

`$this->title` задаём всегда: идёт в `<title>`, заголовок страницы и как
последняя (текущая) крошка.

**Множественное имя `public static $titles`** (plural) — канон для заголовка
списка, ссылки-родителя в крошках и тултипов. Объявляется у КАЖДОЙ модели,
у которой есть страницы (единственное `$title` появилось как стандарт раньше
и остаётся, но для плюрала всегда берётся `$titles`). Базовый `ArmsModel`
даёт дефолт «Объекты», поэтому наличие осмысленного `$titles` стережёт
`tests/unit/models/ModelTitlesTest` — иначе авто-вывод крошек/заголовков
молча показал бы «Объекты».

## 9. Цветовые маркеры (Markers)

Раскраска объектов справочников — НЕ рукописным CSS, а справочником
«Маркеры» (`models/Markers`, admin-меню «шестерёнка»). Маркер = до трёх
цветов, обязателен один:

- `color` — фон, **универсальный канал**: его рендерят все контексты
  (список, тёмная шапка, тултип, карточка, уголок, будущие IPAM/композиты);
  смысл маркера должен читаться по одному фону;
- `border_color`+`border_style` (solid/dashed) — уточняющий канал
  (семантика DMZ-пунктиров); **правило деградации**: где рамку рендерить
  негде (уголок `unit-status`, полоски смешения) — она молча отбрасывается;
- `text_color` — переопределение; по умолчанию автоконтраст к фону
  (`ColorHelper::contrastColor`).

Механика: элемент получает класс `marked-item` + инлайн CSS-переменные
`--marker-bg/-fg/-border` (`Markers::getStyleVars()`); потребляет их ОДИН
базовый файл `web/css/markers.css` — пер-объектных CSS-оверрайдов под
контексты (тёмная шапка и т.п.) больше не пишем: маркер сам задаёт и фон,
и цвет текста.

Подключение модели-владельца: колонка `marker_id` + `MarkerOwnerTrait`
(связь `marker`, хелперы `markerClass($legacy)`/`markerStyle()` для ручных
span/td) + запись в `linksSchema` обеих сторон (обратной `_ids`-стороне
нужна одноимённая relation в Markers — LinkerBehavior её зовёт при save).
`ItemObjectWidget` подхватывает маркер сам (`$model->marker`), параметром
`marker` можно передать чужой (сеть/IP красятся маркером сегмента,
VLAN — маркером L2-домена) или `false` — отключить. Легаси CSS-классы
(`state_*`, `segment_*`, `net-domain-<имя>`) остаются fallback'ом, ТОЛЬКО
пока объекту не назначен маркер, и подлежат выпиливанию.

Выбор маркера в формах — обычный `->select2()`: `ActiveField` сам
распознаёт ссылку на Markers и рисует варианты в целевой раскраске
(`Markers::fetchSelectData()` → data-marker-style на option'ах →
`formatSelect2MarkerItem` в select2hints.js), ttip-хинты сохраняются.

**Производительность (обязательно).** Раскраска строки списка обращается к
`->marker` — без жадной загрузки это N+1 на каждую строку. Любой список,
который рендерит маркированные элементы, обязан дописывать `.marker` в свой
`joinWith`/`with` (`state.marker`, `netIps.network.segment.marker`, …) —
см. TechsSearch/CompsSearch/NetworksSearch и генерик-ветку
`ArmsBaseController::actionIndex` (`with('marker')` для справочников без
Search-модели). Подключили маркер к новой модели — проверьте её списки.

## Шаблоны описаний

Цель — чтобы конкатенации (атрибут + тип) и переходы между слоями читались
как один связный текст. Каждый текст отвечает ровно на свой вопрос и не
залезает на территорию соседа.

### hint атрибута (`attributeData`) — «ЧТО здесь хранится»

- 1–3 предложения, до ~300 символов, допустима лёгкая разметка
  (`<br>`, `<b>`, `<i>`); заголовки/списки разделов — признак, что тексту
  место в слое 2.
- Не описывает формат ввода (его даст `inputHint()` типа), не повторяет label.
- Шаблон: `<Что хранится/означает>. [<На что влияет / когда заполнять>]`

```php
//хорошо: только смысл поля в контексте модели
'hint' => 'MAC адреса сетевых интерфейсов, настроенных в ОС',
//плохо: формат — территория типа, будет продублирован
'hint' => 'MAC адреса. Заполняются по одному в строке',
```

### `modelDescription()` — «ЧТО это за сущность»

- 1–3 предложения: что это за объекты, что учитывается, ключевые связи.
- Без инструкций (они в guides) и перечисления всех полей (оно генерируется).

```php
public static function modelDescription(): string
{
	return 'Единицы оборудования: серверы, АРМ, сетевые устройства, периферия. '
		.'Учитываются модель, серийный/инвентарный номера, размещение, состояние '
		.'и связи с ОС, пользователями и документами.';
}
```

### `inputHint()` типа — «КАК вводить»

- Начинается с существа формата, без вводных («В каждой строке — …»,
  «Формат: …», «Выберите из списка…»).
- Не упоминает конкретные модели/атрибуты — текст один на все места
  использования типа.
- Выводится приглушённо (`text-muted`) после хинта атрибута — вместе они
  читаются как «что это» + «как заполнять».

### `searchHint()` типа — «КАК искать»

- Только отличия/дополнения к общему синтаксису поиска (`|`, `&`, `!` уже
  описаны в дефолтной подсказке, к которой текст подклеивается).

### MD сущности (`docs/help/models/<class-id>.md`)

Только концепция — **без разделов по отдельным атрибутам** (короткое
описание атрибута живёт в hint, длинное — в `models/<class-id>/<attr>.md`;
на странице `/docs/model` справочник атрибутов генерируется, а атрибутные
страницы открываются из него ссылками «подробнее» модалкой).

Канонический скелет (plans/help-inline.md): преамбула-концепция + строгие
опциональные H2 — их точные названия адресуются трансклюзией
(DocsPanelWidget), опечатку ловит сторож `HelpSectionCanonTest`:

```markdown
# <Titles модели>

<Преамбула: что это и зачем, связи — самодостаточное резюме.>   → index-панель, /docs, GitHub

## Список          → index-панель (что в списке, навигация, фильтры)
## Просмотр         → view-панель (композиция карточки: вкладки, блоки)
## Добавление       → только /docs и GitHub
## Редактирование   → только /docs и GitHub
## Удаление         → только /docs и GitHub
```

Все H2 опциональны (после view-hints атрибуты самоописательны — секции
нужны только там, где есть нюанс сверх подсказок полей). Обязательна
только преамбула. Неканонические H2 допустимы, но в инфоблоки не попадают
(видны лишь в /docs и на GitHub). Сценарий одного атрибута
(переименование, архивация) — не секция модельного файла, а
`models/<class-id>/<attr>.md`.

Одиночной канонической секции заголовок в панели не нужен (его даёт подача
страницы), поэтому по умолчанию строка H2 отбрасывается. Но когда панель
собирает **несколько секций гайда** (IPAM: преамбула + «Раскраска» +
«Куда кликать» + «Параметры»), без заголовков они сливаются в сплошной
текст — таким вызовам ставим `'headings'=>true` (H2/H3 внутри `.docs-panel`
приглушены до подзаголовков в `qtip.css`).

### MD атрибута (`docs/help/models/<class-id>/<attr>.md`)

```markdown
# <Titles> → <Label атрибута>

<Абзац: когда и как заполняется, на что влияет.>

## Примеры
## Связанное
<опционально ссылка на страницу типа: [тип …](../../types/<type>.md)>
```

### MD типа (`docs/help/types/<type>.md`)

Правила ввода и поиска — территория **лёгкого** слоя (`inputHint()`/
`searchHint()`), на странице типа они не повторяются. Сюда — то, что
в тултип не влезает: устройство, хранение, ограничения, примеры.

```markdown
# <Человеческое название типа>

<Абзац: что за данные, где используются.>

## Хранение и нормализация
## Особенности и ограничения
## Примеры
```

### Guides / Admin (`docs/help/guides|admin/*.md`)

```markdown
# Как <задача пользователя>   /   # <Процедура администрирования>

<Абзац: результат и когда это нужно.>

1. Шаг…
2. Шаг…
```

Общие правила страниц: первая строка — `# Заголовок` (он же title и подпись
ссылок), первый абзац — самодостаточное резюме (в модалке виден первым),
ссылки между страницами относительные, картинки — только когда без них никак.
