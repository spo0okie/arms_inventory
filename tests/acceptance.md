# Acceptance тесты проекта ARMS

## Назначение и описание

Проверка работоспособности системы с точки зрения конечного пользователя. Они моделируют реальные сценарии использования приложения и проверяют, что ключевые функции работают корректно без ошибок.

## Структура acceptance тестов

```
tests/
├── acceptance.suite.yml          # Конфигурация основных acceptance тестов
├── acceptance-extra.suite.yml    # Конфигурация расширенных acceptance тестов
├── acceptance/                   # Основные acceptance тесты
│   ├── _bootstrap.php            # Инициализация окружения
│   ├── PageAccessCest.php       # Тесты доступности страниц
│   ├── HistoryPagesCest.php     # Тесты страниц истории
│   └── AuthorizationModesCest.php # Тесты режимов авторизации
├── acceptance-extra/             # Расширенные acceptance тесты
│   ├── _bootstrap.php
│   └── PageAccessCest.php       # Расширенные тесты доступности
└── _data/                       # Тестовые данные
    ├── get-routes-data.php      # Конфигурация маршрутов
    └── arms_demo.sql            # Демонстрационная база данных
```

## Существующие acceptance тесты

### PageAccessCest

**Файл**: [`tests/acceptance/PageAccessCest.php`](tests/acceptance/PageAccessCest.php)

#### Назначение

Тест проверяет работоспособность всех методов всех типовых контроллеров UI части приложения. Использует динамическую генерацию тестов на основе обхода контроллеров и их методов и ручного уточнения конфигурации маршрутов.

#### Генеративный механизм

Тест использует **максимально генеративный подход** для минимизации ручной конфигурации:

##### 1. Динамическое сканирование контроллеров

- Автоматически находит все контроллеры в директории `controllers/`
- Фильтрует только наследников `ArmsBaseController`
- Не требует ручной регистрации контроллеров

##### 2. Динамическое получение actions

- Получает actions из конфигурации контроллера
- Использует Reflection для поиска методов `action*()`
- Автоматически обнаруживает новые actions
- Для каждого action генерирует тестовый маршрут
  - Параметры теста для маршрута ищутся через regex из `get-routes-data.php`

##### 3. Система макросов для подстановки данных 

Параметры вызова тестового маршрута могут содержать макросы, которые будут заменены на реальные значения перед выполнением теста.

**Доступные макросы:**

| Макрос | Источник | Описание |
|--------|----------|----------|
| `{anyId}` | `$models[0]->id` | ID первой модели |
| `{otherId}` | `$models[1]->id` | ID второй модели |
| `{anyName}` | `$models[0]->name` | Название первой модели |
| `{anyModelParams}` | Атрибуты модели | Все атрибуты формы первой модели |
| `{otherModelParams}` | Атрибуты модели | Все атрибуты формы второй модели |
| `{replacedModelParams}` | Сохранённая модель | Параметры через `saveModel` |
| `{deletedModelParams}` | Сохранённая модель | Параметры удалённой модели |

##### 4. Поддержка множественных вариантов маршрутов 

- Каждый маршрут может иметь несколько вариантов тестирования 
  (Позволяет тестировать один маршрут с разными параметрами)
- Поддерживает варианты `[1]`, `[2]`, `[3]` и т.д.
- Пример: `update[1]` с другими параметрами, чем `update`

##### 5. Проверка целостности данных 

- Использует `getLinksSchema()` для определения связей
- Удаляет обратные связи (Many-to-Many) перед удалением модели
- Гарантирует целостность данных при тестировании delete

#### Алгоритм работы routesProvider()

Метод `routesProvider()` динамически формирует набор тестовых маршрутов (пути и параметры):

```
routesProvider()
├── 1. Загрузка конфигурации
│   └── require tests/_data/get-routes-data.php (regex-паттерны)
├── 2. Инициализация тестового окружения
│   ├── Helper\Yii2::initFromFilename('test-web.php')
│   ├── Helper\Database::prepareYiiDb()
│   └── Helper\Database::loadSqlDump('arms_demo.sql')
├── 3. Сканирование контроллеров
│   └── scandir(controllers/) → только наследники ArmsBaseController
├── 4. Получение actions
│   ├── $controller->actions() (объявленные в конфиге)
│   └── ReflectionClass: поиск методов action*()
├── 5. Формирование маршрута
│   └── route = "{controllerId}/{actionId}"
├── 6. Поиск параметров в конфиге
│   └── ArrayHelper::findByRegexKey($params, $route)
├── 7. Поддержка вариантов [N]
│   └── Поиск [1], [2]... если базовый маршрут не найден
└── 8. Сортировка
    ├── Контроллеры: по алфавиту
    └── Actions: delete → validate → update → create → view
```

#### Конвейер CRUD операций: Стратегия тестирования целостности данных

**Назначение конвейера**

Порядок сортировки действий (delete → validate → update → create → view) реализует специальный конвейер для тестирования целостности данных и отсутствия коллизий при CRUD операциях. Этот подход гарантирует, что для записи в модель будут использоваться валидные данные, которые не вызовут коллизий с другими моделями.

**Пошаговый процесс конвейера**

1. **delete** — удаление первой модели
   - Загружается первая найденная модель (макрос `{anyId}`)
   - Перед удалением удаляются все обратные связи (Many-to-Many) через `dropReverseLinks()`
   - Параметры модели сохраняются в `$this->savedModels[$modelClass]['deleted']` перед удалением
   - Ожидается ответ 302 (редирект на index)
   - **Результат:** Модель 1 удалена, её параметры сохранены

2. **validate** — проверка валидации параметров удалённой модели
   - Используются сохранённые параметры удалённой модели (макрос `{deletedModelParams}`)
   - Отправляются как POST параметры на action validate
   - Так как модель была удалена, не возникает коллизий при проверке уникальности
   - **Результат:** Валидация работает корректно без коллизий

3. **update** — обновление второй модели параметрами первой
   - Загружается вторая модель (макрос `{otherId}`, отличная от первой)
   - Параметры второй модели сохраняются в `$this->savedModels[$modelClass]['replaced']` перед изменением
   - Отправляются параметры удалённой первой модели (макрос `{deletedModelParams}`)
   - Так как первая модель удалена, не возникает коллизий при сохранении
   - **Результат:** Модель 2 изменена на параметры модели 1, её исходные параметры сохранены

4. **create** — создание новой модели с параметрами изменённой модели
   - Используются сохранённые параметры модели 2 (макрос `{replacedModelParams}`)
   - Отправляются как POST параметры на action create
   - Так как модель 2 была изменена на параметры модели 1, её исходные параметры уникальны
   - **Результат:** Новая модель создана с исходными параметрами модели 2

5. **view** — просмотр созданной модели
   - Проверяется доступность страницы просмотра новой модели
   - **Результат:** Модель успешно отображается

**Гарантии консистентности данных**

Этот конвейер гарантирует:

- **Отсутствие коллизий при валидации:** Параметры удалённой модели не вызывают ошибок уникальности
- **Отсутствие коллизий при обновлении:** Параметры удалённой модели могут быть присвоены другой модели
- **Отсутствие коллизий при создании:** Исходные параметры модели 2 остаются уникальными
- **Целостность связей:** Обратные связи удаляются перед удалением модели, предотвращая ошибки FK
- **Полнота тестирования:** Каждый CRUD action тестируется в контексте реальных данных

#### Роль файла get-routes-data.php

Файл [`tests/_data/get-routes-data.php`](tests/_data/get-routes-data.php) содержит **regex-паттерны** для маршрутов:

| Тип записи | Пример | Описание |
|------------|--------|----------|
| Пропуск | `'/^history\/.*$/' => '{skipTest}'` | Полное игнорирование |
| Кастомные | `'/^acls\/ace-cards$/' => ['GET'=>['id'=>'{anyId}']]` | Специфичные параметры |
| Универсальный | `'/^[a-z0-9\-]+\/view$/' => ['GET'=>['id'=>'{anyId}']]` | Все контроллеры |
| Вариант [N] | `'/^[a-z0-9\-]+\/update\[1\]$/'` | Альтернативные параметры |

**Структура конфигурации:**

```php
'/^[a-z0-9\-]+\/view$/' => [
    'GET' => ['id' => '{anyId}'],           // GET параметры
    'POST' => [...],                        // POST параметры (опционально)
    'response' => 200,                      // Ожидаемый HTTP код (опционально)
    'saveModel' => [...],                   // Сохранение модели для дальнейшего использования
    'dropReverseLinks' => ['id' => '{anyId}'], // Удалить обратные связи (опционально, нужно для DELETE)
]
```

#### Проблемы

- [ ] Обход контроллеров игнорирует модули: Есть как минимум модуль Schedules чьи контроллеры не тестируются.
- [ ] Тестируется только один сценарий view для каждого контроллера (с неясным набором связей у проверяемой модели, т.к. {anyId} ищет просто первую попавщуюся модель)
  - Это частично закрывается костылями изи get-routes-data выбором наиболее репрезентативной модели из тестовой БД
- [ ] Отсутствуют сценарии с герантированно пустыми Many-to-Many связями
- [ ] Отсутствуют сценарии с гарантированно заполненными всеми связями
- [ ] Возможны модели у которых одни связи исключают другие, это не учитывается ни в тестах ни в самих моделях нет специальных указаний на формирование тестовых маршрутов
- [ ] Файл `get-routes-data.php` содержит много ручной конфигурации (костылей) для специальных случаев, что нарушает идею генеративного подхода.

**Зависимость рендера от связей:**

Успешность рендера зависит от количества и типа связей отображаемой модели:

| Сценарий | Описание | Нужно тестировать |
|----------|---------|------------------|
| Без связей | Модель отображается с заглушками, которые сами могут выдать ошибку | Да |
| С полными связями | Инклюдятся рендеры дополнительных компонент, каждая может выдать ошибку | Да  |
| С частичными связями | Некоторые связи есть, некоторые нет, может быть нарушена целостность | Не обязательно  (текущее) |


### HistoryPagesCest

**Файл**: [`tests/acceptance/HistoryPagesCest.php`](tests/acceptance/HistoryPagesCest.php)

#### Назначение

Тест проверяет доступность страниц истории изменений для всех моделей, имеющих историю.

#### Ключевые особенности

- **Динамическое обнаружение**: Автоматически находит все классы с суффиксом `*History`
- **Обработка ошибок**: Пропускает тесты для отсутствующих таблиц истории
- **Проверка представлений**: Проверяет доступность представлений истории в БД

#### Процесс работы

1. Сканирует директорию `models/` на наличие классов `*History`
2. Для каждого класса проверяет существование таблицы в БД
3. Проверяет доступность страницы истории `/history/view?id=...`
4. При отсутствии таблицы — пропускает тест с сообщением

### AuthorizationModesCest

**Файл**: [`tests/acceptance/AuthorizationModesCest.php`](tests/acceptance/AuthorizationModesCest.php)

#### Назначение

Тест проверяет корректную работу различных режимов авторизации и RBAC (Role-Based Access Control).

#### Режимы авторизации

| Тест | RBAC | Требование авторизации | Описание |
|------|------|----------------------|----------|
| `testModeNoRbacNoAuthRequired` | Выкл | Нет | Без RBAC, без авторизации |
| `testModeRbacEnabledNoAuthRequired` | Вкл | Нет | RBAC включен, авторизация не требуется |
| `testModeRbacEnabledAuthRequired` | Вкл | Да | RBAC включен, авторизация требуется |
| `testModeNoRbacAuthRequired` | Выкл | Да | Без RBAC, авторизация требуется |

#### Реализация

Каждый тест выполняет следующие проверки:

1. Настраивает соответствующий режим авторизации
2. Выполняет запрос к защищённой странице
3. Проверяет корректность поведения:
   - При включённой авторизации — редирект на страницу входа
   - При выключенной авторизации — доступ к странице разрешён
   - При включённом RBAC — проверка прав доступа

## Конфигурация тестового окружения

### Структура конфигурации

#### PhpBrowser модуль

Модуль `PhpBrowser` используется для эмуляции HTTP-запросов без запуска полного браузера. Он:

- Выполняет реальные HTTP-запросы к приложению
- Поддерживает сессии и cookies
- Позволяет проверять ответы и элементы страниц

#### Helper модули

- **Yii2 Helper**: Интеграция с Yii2-приложением, поддержка AJAX-запросов
- **Database Helper**: Управление тестовыми данными, очистка БД между тестами

### Требования к окружению

1. **Веб-сервер**: Nginx или Apache на порту 8081
2. **Точка входа**: `web/index-test.php`
3. **Конфигурация БД**: `config/test-db.php`
4. **Демо-данные**: Загруженные из `tests/_data/arms_demo.sql`

## Запуск acceptance тестов

### Основные команды

```bash
# Запуск всех acceptance тестов
vendor/bin/codecept run acceptance

# Запуск расширенных acceptance тестов
vendor/bin/codecept run acceptance-extra

# Запуск конкретного теста
vendor/bin/codecept run acceptance PageAccessCest

# Запуск конкретного метода теста
vendor/bin/codecept run acceptance "PageAccessCest:testPageAccess"

# Запуск тестов истории
vendor/bin/codecept run acceptance HistoryPagesCest

# Запуск с детальным выводом
vendor/bin/codecept run acceptance --debug
```

### Опции запуска

| Опция | Описание |
|-------|----------|
| `--debug` | Подробный вывод отладочной информации |
| `--steps` | Пошаговый вывод выполнения тестов |
| `--filter` | Фильтр по названию теста |
| `--group` | Запуск тестов определённой группы |
| `--report` | Генерация HTML-отчёта |

## Известные проблемы и решения

### 7.1 PHP 8.2 deprecated warnings

**Проблема**: PHP 8.2 генерирует предупреждения о deprecated-функциях в Yii2 и связанных библиотеках.

**Решение**: Подавлены в `config/test-web.php`:

```php
error_reporting(E_ALL & ~E_DEPRECATED & ~E_USER_DEPRECATED);
```
