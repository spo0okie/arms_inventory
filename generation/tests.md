# План работ по генеративным acceptance-тестам

## Цель
Перейти от ручных acceptance-наборов к генеративному механизму, где:
- сценарии описываются в контроллерах и/или связанных провайдерах;
- для каждого `action*` существует отдельный `test*` DataProvider;
- тест собирает все контроллеры, их методы и соответствующие DataProvider;
- текущий `tests/acceptance` полностью заменяем и может быть удалён.

## Этапы
1. Зафиксировать формат описания сценариев в документации
   - Обновить `generation/readme.md`: определить формат провайдера сценариев.
   - Указать, что `route` по умолчанию = `{controllerId}/{action}` (не пишется в сценариях).
   - Описать обязательные и опциональные ключи сценария.

2. Ввести базовый провайдер сценариев в `ArmsBaseController`
   - Добавить контракт для сценариев на уровне базового контроллера.
   - Базовая логика: отдельный `test*` метод на каждый `action*`.
   - Механизм наследования/переопределения.

3. Реализовать генеративный тест
   - Новый Cest/тест, который:
     1) находит все контроллеры
     2) извлекает `action*`
     3) для каждого `action*` ищет `test*` DataProvider
     4) генерирует сценарии и прогоняет их
   - Применяет значение `route` по умолчанию
   - Учитывает `skip/only/role/expect` и макросы параметров

4. Довести покрытие сценариями до уровня текущих acceptance
   - Мигрировать все ключевые сценарии из `tests/acceptance.md` в генеративный формат.
   - Отразить проблемные зоны: CRUD-конвейер, макросы, связи.

5. Прогон тестов
   - Запустить соответствующий suite.
   - Исправить обнаруженные проблемы.

6. Удаление legacy acceptance
   - После успешного прохождения — удалить `tests/acceptance` и `tests/acceptance.md`.
