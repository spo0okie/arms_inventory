## Реализованные компоненты

### Генераторы атрибутов (Этап 1) ✅

Создан базовый слой генераторов значений для всех используемых в проекте типов атрибутов, кроме типа "связь", который требует генерации моделей.

### Интерфейс генератора

```php
interface GeneratorInterface
{
    public static function generate(array $params): mixed;
}
```

### GeneratorResolver

Класс `GeneratorResolver` отвечает за выбор генератора:

- По типу атрибута из `attributeData()`
- Поддержка кастомных генераторов через `generator` в attributeData
- Маппинг типов → генераторы

### Тестирование

Создан unit-тест `GeneratorResolverTest` проверяющий покрытие всех типов атрибутов моделей генераторами

## Следующие этапы

### Этап 2: ModelFactory ✅ РЕАЛИЗОВАН

Создан класс [`ModelFactory`](generation/ModelFactory.php:1):

```php
use app\generation\ModelFactory;

// Модель с заполненными атрибутами (по умолчанию)
$model = ModelFactory::create(ModelClass::class);

// Модель с пустыми атрибутами (nullable)
$model = ModelFactory::create(ModelClass::class, [], [
    'empty' => true,
]);

// С переопределениями атрибутов
$model = ModelFactory::create(ModelClass::class, [], [
    'overrides' => [
        'name' => 'Тестовое имя',
        'model_id' => 123,  // Связь нужно указать вручную
    ],
]);

// С preset для связей (если модель поддерживает roles())
$model = ModelFactory::create(Techs::class, [], [
    'role' => 'pc',  // Требует реализации static function roles() в модели
]);
```

**Возможности:**
- Pipeline: instantiate → generate attributes → apply presets → validate → save
- Retry логика при валидации и сохранении (до 7 попыток)
- Сценарии: `empty` — пустая/заполненная модель
- Presets (роли) — требуют реализации в модели через `static function roles()`
- Связи (`_id`, `_ids`) — указываются вручную через `overrides`
- Автоматический обход типичных ошибок валидации

## Решаемая проблема

Текущий механизм acceptance тестов хоть и пытается быть генеративным, но по сути опирается на фиксированные Mock данные которые нужно постоянно обновлять. Поэтому тесты всегда запаздывают за кодом и часть проблем пролетает через тестирование в прод.

Цель: перейти при прохождении acceptance тестов от тестовой предазполненной Mock данными БД к возможности создания моделей на лету по запросу. Модели вовсе не обязательно должны содержать сколь нибудь осмысленные данные, т.к. вся суть acceptance тестов - провека собирается ли страница или падает (ошибка 500)

Что это дает:

- Возможность проверить форму отображения модели полностью заполненную данными
  - Включая каскадное заполнение данных у связанных моделей
- Возможность проверить форму отображения модели полностью без данных (где это допустимо)
- Возможность проверить поиск подели по атрибутам (мы заранее легко можем создать модель с нужными атрибутами)

## Описание

Тут наверно будет ларь с ценными мыслями, не все из которых доживут до релиза

### ModelFactory

В конечном итоге нам надо собрать класс ModelFactory, который сможет создавать модели по запросу. 
Для этого ему нужно

- сгенерировать значение для каждого атрибута -> AttrbiteGenerator
- проверить cross-fields сценарии -> ModelResolver

ModelFactory pipeline:

1. instantiate model
2. generate attributes (AttributeGenerator)
3. apply presets (role), overrides
4. apply ModelResolver
5. validate
6. retry if fail
7. save
8. retry if fail

### AttributeGenerator

Входные данные

- Модель для получения данных об атрибутах (тип, required, unique)
- Целевой статус атрибута: пока представляю себе
  - пустой / null (в проекте придерживаемся подхода null эквивалентен пустому)
  - заполненный минимально (может быть все равно заполнен в случае required)
  - заполненный полностью (с указанием глубины связей полного заполнения)

Позже можно добавить для наследуемых атрибутов, но это чревато взрывом комбинаций

- Пустой без предка
- Пустой с предком без атрибута
- Пустой с предком с заполненным атрибутом
- Заполненный (наличие предка не важно)

Результат

- Возвращает значение, которое надо будет поместить в новую модель, которую передать в ModelResolver

Процесс

Из класса вытаскивает метаданные атрибута

- тип (один из стандартых: urllist/text/string/ips/...) для которых надо сделать генераторы значений
- is_unique/is_required надо вытаскивать из валидаторов модели
- проверяет какие-то типовые параметры из валидаторов (границы длинны строки / границы числовых значений)

### ModelResolver

В Модели аналогично rules для валидатора надо будет писать pipeline из методов коррекции данных для пост-обработки модели. Также скорее всего надо будет писать стандартные резолверы и кастомные.

ModelResolver pipeline:

1. Attribute-level fixes (null/required)
2. Relation fixes (FK, связи)
3. Domain fixes (бизнес-логика)

Делать ли тест, что у модели для всех атрибутов ясно как должен вести себя резолвер? - нет, это не выводится никаким образом. Надо просто обеспечить модель необходимым набором резолверов, чтобы созданная модель проходила валидацию. Такого теста достаточно.

### Types

создать в корне проекта папку types (namespace app/types) и в ней создавать классы типов:

```php
interface AttributeType {
  public static function generate($model,$attr);
}

class String () {
  public static function generate($model,$attr) {
    //вытаскивает min-max из rules и заполняет строку длинной в нужных пределах alpha-numeric
  }
}
```

### Retries

1. generate
2. validate
3. если fail → retry (нужно ограничить число попыток validate retry)
4. save
5. если fail → retry (нужно ограничить число попыток save retry)

### Всегда ли создавать модели под тест или искать в БД

Создавать, т.к. иначе тесты зависят друг от друга. Это усложнит отладку

### Признак заполняемости

Как выяснить что атрибут заполняется при записи в модель?

- есть readOnly атрибуты
- есть атрибуты которые заполняются onBeforeSave
- есть атрибуты связи моделей, которые заполняются только у одной из моделей (добавляем производителя в софт но не наоборот)

кажется надежный способ

- присутствует в attributes()
- присутствует в rules() (safe)
- НЕ является:
  - PK (обычно)
  - auto-increment
  - вычисляемым

### Описание acceptance тестов

Имеет смысл перейти к описанию их прямо в контроллерах для каждого action
Это позволит

- наследовать типовые проверки от базового контроллера
- кастомизировать, дополнять базовые тесты для сложных сценариев
- обнаруживать у контроллера actions без тестов

### Random

Если не делать посев (seed), то результаты могут меняться от прогона к прогону. Нужны воспроизводимые тесты

- либо всегда один seed (начать наверно проще с этого)
- либо seed теста сохраняется и можно воспроизвести тест с тем же seed

В любом случае seed лучше логировать перед каждым тестом, чтобы его можно было воспроизвести

## Шаги

### Типизация атрибутов

Для того чтобы у модели можно было сгенерировать атрибуты нужны генераторы, который смогут выдавать валидные данные на основании типов. Для этого у каждого атрибута как минимум однозначно должен идентифицироваться тип.

Нужно сделать тест, который будет проверять у всех моделей закрытие вопросы типизации "заполняемых" атрибутов

### Типовые генераторы

Нужны линейные генераторы валидных значений для каждого типа

### Нетиповые генераторы

#### Генератор Model Preset

При заполнения поля оборудования arms_id надо указывать ссылку на другое оборудование, но принципиально модели из типа ПК.
Т.е. сценарий заполнить атрибут arms_id -> создать Techs с заполненным атрибутом model_id -> создать TechModels с указанным tech_type_id -> создать TechTypes с проставленным is_comp= true;

Как этот сценарий описать в генераторе?:
Можно добавить в linkSchema preset для связи например

```php
  public $linksSchema=[
    'arm_id' => [Techs::class,'preset'=>'pc'],
  ];
```

А в самом Techs

```php
class Tech extends ArmsModel {
  public static function roles(): array {
    return [
      'pc' => function($m) {
        $type = ModelFactory::create(TechType::class, [
          'is_comp' => true
        ]);

        $model = ModelFactory::create(TechModel::class, [
          'tech_type_id' => $type->id
        ]);

        $m->model_id = $model->id;
      }
    ]
  }
}
```

Preset обязан:

- возвращать валидную модель
- быть идемпотентным
